## 🔍 核心问题诊断

### 问题 1：类型断言泛滥 ⭐⭐⭐⭐⭐

**严重度**：高  
**影响范围**：21 处

**典型案例**：
```typescript
// app/(admin)/posts/posts-content.tsx:79
<TableCell>{(post.category as { name?: string })?.name ?? '-'}</TableCell>

// app/blog/[slug]/page.tsx:202-215
{(post.author as { name?: string })?.name || 
 (post.author as { username?: string })?.username}
```

**问题根源**：
- 数据库查询返回的关系类型不精确
- 缺少明确的 `PostWithRelations` 类型定义
- 类型推断依赖 Drizzle ORM 的自动推断

**影响**：
- ❌ 类型安全性降低
- ❌ 代码可读性差
- ❌ 容易出现运行时错误
- ❌ IDE 智能提示失效

---

### 问题 2：组件职责不清 ⭐⭐⭐⭐

**严重度**：中高  
**影响范围**：多个大型组件

**典型案例**：

#### `app/write/write-form.tsx` (418 行)
```typescript
// 混合了以下职责：
1. 表单状态管理（useState × 5）
2. 自动保存逻辑（useEffect + localStorage）
3. Slug 自动生成
4. 表单验证（react-hook-form）
5. 提交处理（useTransition）
6. UI 渲染（复杂的布局和侧边栏）
```

#### `components/blog/post-form.tsx` (287 行)
```typescript
// 与 write-form.tsx 有 70% 重复代码
1. 相同的表单字段
2. 相同的验证逻辑
3. 相同的提交处理
4. 仅编辑/创建模式不同
```

**问题根源**：
- 未抽离业务逻辑到 Hooks
- 未拆分 UI 组件
- 未复用表单字段组件

**影响**：
- ❌ 代码重复率高（70%）
- ❌ 维护成本高（改一处需要改两处）
- ❌ 测试困难
- ❌ 难以扩展

### 问题 4：数据获取模式混乱 ⭐⭐⭐⭐

**严重度**：中高  
**影响范围**：3 处 fetch + 22 处 Server Actions

**当前模式**：

#### 模式 A：Server Actions（主流）
```typescript
// 22 处使用
import { createPost, updatePost } from '@/lib/actions/posts'
await createPost(data)
```

#### 模式 B：API Routes + fetch（少量）
```typescript
// 3 处使用
// 1. 注册功能
const response = await fetch('/api/auth/register', { method: 'POST' })

// 2. 无限滚动
const response = await fetch(`/api/posts?${params}`)

// 3. 预加载
await fetch(`/api/posts?${params}`, { priority: 'low' })
```

**问题根源**：
- 架构决策不一致
- 注册功能不必要地使用 API Route
- TanStack Query 已安装但完全未使用（浪费 500KB）

**影响**：
- ❌ 架构不统一
- ❌ 开发者困惑
- ❌ 手动状态管理复杂（149 行）
- ❌ 无缓存、去重、重试机制

---

### 问题 5：缺少数据访问层 ⭐⭐⭐⭐

**严重度**：中  
**影响范围**：Server Actions 混合查询和业务逻辑

**典型案例**：
```typescript
// lib/actions/posts.ts
export async function getPostBySlug(slug: string) {
  // 直接在 action 中写查询逻辑
  const post = await db.query.posts.findFirst({
    where: eq(posts.slug, slug),
    with: {
      author: { columns: { ... } },
      category: true,
      tags: { with: { tag: true } }
    }
  })

  // 混合了数据转换逻辑
  return {
    ...post,
    tags: post.tags.map(pt => pt.tag)
  }
}
```

**问题根源**：
- 未采用 Repository Pattern
- 查询逻辑和业务逻辑耦合
- 数据转换分散在各处

**影响**：
- ❌ Server Actions 代码臃肿（平均 152 行）
- ❌ 查询逻辑难以复用
- ❌ 难以单元测试
- ❌ 类型转换不统一

---

### 问题 6：Hooks 未充分利用 ⭐⭐⭐

**严重度**：中  
**影响范围**：16 处 React Hooks 使用分散在组件中

**当前状态**：
- 已有 9 个通用 Hooks（use-debounce, use-media-query 等）
- 仅 2 个业务 Hooks（use-infinite-posts, use-intersection-observer）
- 组件中有大量重复的状态管理逻辑

**典型案例**：
```typescript
// components/blog/like-button.tsx (57 行)
const [isPending, startTransition] = useTransition()
const [liked, setLiked] = useState(initialLiked)
const [count, setCount] = useState(initialCount)

const handleToggleLike = () => {
  startTransition(async () => {
    try {
      const result = await togglePostLike(postId)
      setLiked(result.liked)
      setCount(result.likesCount)
    } catch (error) {
      // 错误处理
    }
  })
}

// components/blog/bookmark-button.tsx (55 行)
// 几乎相同的逻辑，只是调用不同的 action
```

**问题根源**：
- 未抽离业务逻辑到 Hooks
- 未创建通用的交互 Hooks
- 未统一错误处理

**影响**：
- ❌ 代码重复（like-button 和 bookmark-button 80% 相似）
- ❌ 逻辑分散，难以维护
- ❌ 错误处理不一致
- ❌ 难以测试
